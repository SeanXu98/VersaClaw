## 一、项目概述

### 1.1 项目背景

**Nanobot-Swarm** 是基于香港大学数据科学学院（HKUDS）开源项目 Nanobot 的二次开发项目，旨在为 Nanobot 构建一个功能完整的可视化 Web 管理平台。该项目通过现代化的前后端分离架构，实现了对 AI Agent 的全面管理和实时交互。

### 1.2 核心价值

- **统一 LLM 接入**：支持 15+ 主流 LLM 提供商（OpenAI、Anthropic、DeepSeek、通义千问、智谱 GLM、Kimi 等）
- **多渠道消息接入**：支持 9 种即时通讯平台（Telegram、Discord、Slack、飞书、钉钉、QQ、WhatsApp、Email、Mochat）
- **可视化 Agent 管理**：提供模型管理、实时聊天、技能编辑、记忆管理等 Web 界面
- **流式交互体验**：基于 SSE 的实时流式聊天，可视化展示 Agent 思考过程和工具调用

### 1.3 技术栈概览

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| **前端框架** | Next.js 16 / React 19 | 现代化 Web 前端 |
| **前端语言** | TypeScript 5.9 | 类型安全的 JavaScript |
| **前端样式** | Tailwind CSS 3.4 | 原子化 CSS 框架 |
| **后端框架** | Python 3.11+ / FastAPI | 高性能异步 Web 框架 |
| **LLM 统一接口** | LiteLLM | 统一 15+ LLM 提供商的 API 调用 |
| **数据验证** | Pydantic v2 | 强类型配置和数据校验 |
| **异步编程** | asyncio | 原生异步支持 |
| **日志系统** | Loguru | 结构化日志记录 |
| **数据存储** | JSON / JSONL / Markdown | 轻量级文件存储 |
| **容器化** | Docker / Docker Compose | 一键部署前后端服务 |

### 2.2 设计哲学

**1. 最小依赖原则**

核心功能仅依赖 Python 标准库和少量必要库，避免沉重的框架绑定。这使得项目易于部署和维护，同时保持了良好的可移植性。

**2. 文件即数据库**

采用 JSON、JSONL、Markdown 等纯文本格式存储数据。这种设计有三个显著优势：便于调试（可直接查看和编辑）、支持版本控制（Git 友好）、易于迁移（无数据库依赖）。

**3. 异步优先**

全面采用 Python 原生异步编程（asyncio），确保高并发场景下的性能表现。所有 I/O 操作（网络请求、文件读写）都是异步的，避免了阻塞等待。

**4. 统一抽象**

无论是 LLM 提供商、聊天渠道还是工具，都通过抽象基类实现统一接口。这种设计使得扩展新功能变得标准化和简单化。

## 三、Nanobot 核心模块详解

### 3.1 AgentLoop - 智能体核心引擎

AgentLoop 是整个系统的大脑，它的工作原理类似于人类处理问题的思维过程。

#### 运行机制

当一个用户消息到达时，AgentLoop 执行以下步骤：

1. **会话获取**：从会话管理器中获取或创建该用户的会话上下文
2. **上下文构建**：通过 ContextBuilder 将历史对话、长期记忆、技能文档等信息组装成完整的提示词
3. **LLM 调用**：调用大语言模型生成响应
4. **工具执行**：如果 LLM 决定使用工具，执行相应的工具调用
5. **迭代循环**：将工具返回的结果再次喂给 LLM 继续思考
6. **响应返回**：当 LLM 给出最终答案或达到最大迭代次数时结束

#### 迭代循环设计

AgentLoop 的迭代循环设计是其核心创新点。传统的聊天机器人通常只进行一次 LLM 调用就返回结果，而 AgentLoop 允许 LLM 进行多轮"思考-行动"迭代。

例如，当用户问"帮我分析今天的服务器日志"时：

1. LLM 首先调用 `list_dir` 查看日志目录
2. 然后调用 `read_file` 读取具体日志文件
3. 可能再调用 `exec` 执行分析脚本
4. 最后综合所有信息给出分析报告

整个过程是自动的，无需用户干预。

```
┌─────────────────────────────────────────────────────────────┐
│                    Agent 迭代循环                            │
│                                                              │
│     构建消息  ───▶  调用 LLM  ───▶  解析响应                 │
│                        │                                     │
│              ┌─────────┴─────────┐                           │
│              ▼                   ▼                           │
│         有工具调用？          无工具调用                      │
│              │                    │                          │
│              ▼                    ▼                          │
│         执行工具            返回最终响应                      │
│              │                                               │
│              ▼                                               │
│         添加工具结果                                          │
│         到消息列表                                            │
│              │                                               │
│              ▼                                               │
│         继续下一轮？  ─── Yes ──▶  返回循环开始               │
│              │                                               │
│              No                                              │
│              ▼                                               │
│         返回响应                                              │
└─────────────────────────────────────────────────────────────┘
```

#### 流式处理实现

为了提供实时的交互体验，AgentLoop 实现了流式处理接口。它将 LLM 的流式输出、工具调用过程、推理过程等封装成统一的事件流，通过 Server-Sent Events 技术实时推送到前端，让用户能够看到 Agent 的"思考过程"，大大提升了交互的透明度和信任感。

### 3.2 MessageBus - 异步消息总线

MessageBus 实现了"渠道"与"智能体核心"之间的完全解耦，采用了经典的生产者-消费者模式。

#### 设计原理

在传统架构中，如果要让一个聊天机器人同时支持 Telegram、Discord、飞书等多个平台，通常需要在每个平台的处理逻辑中直接调用 AI 处理代码，导致代码高度耦合。

MessageBus 的创新在于引入了一个中间层：

- 所有渠道只需将收到的消息"投递"到入站队列
- 所有响应只需"投递"到出站队列
- AgentLoop 只需从入站队列"消费"消息、将响应"发布"到出站队列

```
┌─────────────────────────────────────────────────────────────┐
│                      MessageBus                              │
│                                                              │
│   入站队列                         出站队列                   │
│  ┌─────────────┐                 ┌─────────────┐            │
│  │ InboundMsg  │                 │ OutboundMsg │            │
│  │   Queue     │                 │   Queue     │            │
│  └─────────────┘                 └─────────────┘            │
│        ▲                               │                     │
│        │                               ▼                     │
│  ┌─────────────────────────────────────────────────────┐    │
│  │                   订阅分发器                          │    │
│  │  {"telegram": [callback1], "discord": [callback2]}  │    │
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

#### 消息流转过程

1. 用户在 Telegram 发送消息
2. Telegram Channel 组件将消息封装成统一的 InboundMessage 对象
3. 消息被投递到入站队列
4. AgentLoop 持续监听入站队列，收到消息后进行处理
5. 生成 OutboundMessage 投递到出站队列
6. 出站分发器监听出站队列，根据消息的 channel 字段路由到对应渠道发送

### 3.3 ToolRegistry - 工具注册与执行系统

工具系统是 AI Agent 与外部世界交互的桥梁。

#### 工具抽象设计

每个工具都必须实现统一的抽象接口，包含四个核心要素：

- **名称**：用于 LLM 调用的唯一标识符
- **描述**：告诉 LLM 这个工具的功能和使用场景
- **参数模式**：JSON Schema 格式的参数规范
- **执行方法**：实际的业务逻辑实现

这种设计使得工具的定义标准化，LLM 可以通过统一的方式理解和使用所有工具。

#### 内置工具矩阵

| 工具类别 | 工具名称 | 功能描述 |
|---------|---------|---------|
| **文件系统** | read_file | 读取文件内容 |
| | write_file | 写入文件内容 |
| | edit_file | 编辑替换文本 |
| | list_dir | 列出目录内容 |
| **Shell 执行** | exec | 执行 Shell 命令 |
| **Web 工具** | web_search | Brave 搜索引擎 |
| | web_fetch | 获取网页内容 |
| **消息工具** | message | 发送跨渠道消息 |
| **子代理** | spawn | 创建子 Agent 执行后台任务 |
| **定时任务** | cron | 创建/管理定时任务 |

#### 工具执行流程

1. LLM 返回工具调用请求
2. ToolRegistry 根据工具名称找到对应的工具实例
3. 根据 JSON Schema 对参数进行校验
4. 异步执行工具的 execute 方法
5. 执行结果以字符串形式返回
6. 结果被添加到对话历史中供 LLM 下一轮迭代使用

#### 安全机制

- **工作空间限制**：文件系统工具支持限制访问指定目录内的文件
- **命令超时**：Shell 执行工具支持超时控制，防止长时间运行命令阻塞系统
- **参数校验**：所有工具参数都经过严格的 JSON Schema 校验

### 3.4 MemoryStore - 三级记忆系统

记忆系统是让 AI Agent 具备"长期记忆"能力的关键，Nanobot 采用了模仿人类记忆的三级架构。

```
┌─────────────────────────────────────────────────────────────┐
│                     三级记忆系统                              │
│                                                              │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 短期记忆 (Session Messages)                          │    │
│  │ • 会话消息历史 (JSONL 格式)                          │    │
│  │ • 保留最近 N 条消息 (memory_window=50)               │    │
│  │ • 实时读写，支持上下文连续                            │    │
│  └─────────────────────────────────────────────────────┘    │
│                          │                                   │
│                          │ 超过窗口大小时触发整合              │
│                          ▼                                   │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 长期记忆 (MEMORY.md)                                 │    │
│  │ • 永久存储的用户偏好、项目背景、重要事实              │    │
│  │ • Agent 主动写入或自动整合                          │    │
│  │ • 每次对话作为系统提示的一部分                       │    │
│  └─────────────────────────────────────────────────────┘    │
│                          │                                   │
│                          ▼                                   │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 历史日志 (HISTORY.md)                                │    │
│  │ • 关键事件和决策的 grep 可搜索日志                    │    │
│  │ • 时间戳标记的历史条目                               │    │
│  │ • 用于回溯过去的对话事件                             │    │
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

#### 短期记忆（会话历史）

短期记忆存储在 JSONL 格式的会话文件中，每条消息作为独立的一行存储。这种设计有两个优势：追加写入效率高，便于流式读取和调试。当对话历史超过配置的"记忆窗口"大小时，旧消息不会直接删除，而是触发记忆整合流程。

#### 长期记忆（MEMORY.md）

长期记忆是一个 Markdown 文件，存储了用户偏好、项目背景、重要决策等持久化信息。每次对话时，这个文件的内容会作为系统提示的一部分发送给 LLM，使 Agent 能够"记住"之前的交互关键信息。

#### 历史日志（HISTORY.md）

历史日志是一个按时间顺序记录重要事件的文件，支持 grep 搜索。当需要回顾过去的某个事件或决策时，可以通过关键词搜索快速定位。

#### 记忆整合算法

当会话消息超过阈值时，系统会在后台异步执行记忆整合：

1. 提取待整合的消息
2. 构建特殊提示词让 LLM 分析这段对话
3. 提取出"值得记住的信息"更新长期记忆
4. 生成"事件摘要"追加到历史日志

这个设计巧妙地利用 LLM 自身的能力来管理记忆，避免了复杂的规则系统。

### 3.5 SessionManager - 会话管理

会话管理器负责维护用户与 Agent 的对话状态，是上下文连续性的基础。

#### 会话标识设计

每个会话通过"渠道:聊天ID"的组合来唯一标识。例如：

- `telegram:12345` - 来自 Telegram 某个用户的会话
- `web:session_abc` - 来自 Web 界面的会话
- `discord:channel_789` - 来自 Discord 某个频道的会话

这种设计确保了跨渠道的会话隔离，同一用户在不同渠道的对话互不干扰。

#### 会话存储格式

会话数据采用 JSONL 格式存储：

- **第一行**：元数据（创建时间、最后整合位置等）
- **后续行**：每行是一条消息记录

这种格式有三大优势：

1. 追加写入只需在文件末尾添加，无需重写整个文件
2. 读取时可以逐行解析，内存友好
3. 纯文本格式便于版本控制和人工检查

#### 缓存与持久化

会话管理器在内存中维护了一个会话缓存（LRU 策略），避免频繁的磁盘读取。当获取会话时，先检查缓存，缓存未命中再从磁盘加载。

### 3.6 ContextBuilder - 上下文构建器

上下文构建器负责将分散的信息组装成完整的提示词。

#### 系统提示词结构

完整的系统提示词包含多个部分：

```
┌─────────────────────────────────────────────────────────────┐
│                    完整系统提示词                            │
│                                                              │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 基础指令                                             │    │
│  │ • Agent 角色定义和行为准则                           │    │
│  │ • 当前时间、操作系统、工作目录等环境信息             │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 启动文件 (AGENTS.md, USER.md, TOOLS.md...)          │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 长期记忆 (MEMORY.md)                                 │    │
│  │ • 用户偏好、项目背景、重要事实                        │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 技能摘要                                             │    │
│  │ • 可用技能的列表和简介                               │    │
│  │ • 按需通过 read_file 加载完整文档                    │    │
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

#### 渐进式技能加载

技能存放在工作空间的 skills 目录下，每个技能是一个 Markdown 文件。为了节省 Token，系统采用了"渐进式加载"策略：

- 启动时只将技能的标题和摘要加入提示词
- 当 Agent 需要使用某个技能时，通过 `read_file` 工具读取完整的技能文档

这种设计在保证功能完整性的同时，大大减少了每次请求的 Token 消耗。

## 四、LLM Provider 系统

### 4.1 Provider Registry - 统一提供商注册表

LLM 提供商的多样性是当前 AI 生态的显著特点，Nanobot 通过 Provider Registry 实现了"一次接入，处处可用"。

#### 提供商分类体系

| 类型 | 提供商 | 模型前缀 | 特点 |
|------|-------|---------|------|
| **网关型** | OpenRouter | openrouter/ | 可路由任意模型，sk-or- 前缀 |
| | AiHubMix | openai/ | OpenAI 兼容网关 |
| | Custom | openai/ | 自定义 OpenAI 兼容端点 |
| **国际主流** | Anthropic | 无 | 原生 Claude 支持 |
| | OpenAI | 无 | 原生 GPT 支持 |
| | DeepSeek | deepseek/ | 国产推理模型 |
| | Gemini | gemini/ | Google 大模型 |
| **国内厂商** | DashScope | dashscope/ | 阿里通义千问 |
| | Zhipu | zai/ | 智谱 GLM |
| | Moonshot | moonshot/ | Kimi 大模型 |
| | MiniMax | minimax/ | MiniMax 模型 |
| **本地部署** | vLLM | hosted_vllm/ | 本地部署模型 |

#### 智能匹配机制

当用户配置一个模型名称时，系统会自动匹配最合适的提供商：

1. **关键词匹配**：检查模型名称中的关键词（如 "qwen" 匹配 DashScope）
2. **API Key 前缀检测**：如 "sk-or-" 匹配 OpenRouter
3. **回退策略**：使用配置中第一个有 API Key 的提供商

#### 环境变量管理

不同提供商需要不同的环境变量。系统在每次 LLM 调用前会动态设置正确的环境变量，确保 LiteLLM 能够正确路由请求。

### 4.2 LiteLLMProvider - 统一调用层

#### 模型名称解析

LiteLLM 使用前缀来区分不同提供商的模型。Nanobot 的 LiteLLMProvider 负责将用户配置的模型名称转换为 LiteLLM 所需的格式。

#### 流式响应处理

流式调用比普通调用复杂得多。LLM 的流式响应是逐 token 返回的，工具调用的参数也是逐步累积的。LiteLLMProvider 实现了完整的事件解析逻辑，将原始的流式数据转换为统一的事件格式：

| 事件类型 | 说明 |
|---------|------|
| content | 文本内容片段 |
| reasoning | 推理过程（DeepSeek-R1 等） |
| tool_call | 工具调用信息 |
| done | 流式完成 |

#### 推理模型支持

以 DeepSeek-R1 为代表的推理模型会在响应中包含"思考过程"（reasoning_content）。LiteLLMProvider 特别处理了这种新类型的输出，使其能够正确传递到上层并被展示给用户。

## 五、多渠道接入系统

### 5.1 BaseChannel - 渠道抽象基类

#### 统一接口设计

所有渠道都实现统一的接口：

- **start**：启动监听
- **stop**：停止监听
- **send**：发送消息

这种抽象使得添加新渠道变得标准化——只需关注特定平台的 API 细节，无需理解整个系统。

#### 权限控制机制

每个渠道都支持"允许列表"配置，可以限制哪些用户或群组可以使用机器人。当收到消息时，Channel 会先检查发送者是否在允许列表中（空列表表示公开访问），只有授权用户的消息才会被转发到 MessageBus。

#### 消息标准化

不同平台的消息格式差异很大，BaseChannel 的 `_handle_message` 方法负责将各种格式的消息标准化为统一的 InboundMessage 对象，包含：

- 渠道名称
- 发送者 ID
- 聊天 ID
- 消息内容
- 媒体附件

### 5.2 ChannelManager - 渠道协调器

#### 生命周期管理

ChannelManager 负责所有渠道的启动和停止。它按照配置动态加载启用的渠道，每个渠道作为独立的异步任务运行。这种设计确保了单个渠道的问题不会影响其他渠道的运行。

#### 出站消息路由

当 AgentLoop 生成响应后，响应被投递到 MessageBus 的出站队列。ChannelManager 运行一个后台分发任务，持续监听出站队列，根据消息的 channel 字段将响应路由到对应的渠道发送器。

### 5.3 支持的渠道

| 渠道 | 协议 | 特点 |
|------|------|------|
| Telegram | HTTP Bot API | 官方 Bot API |
| Discord | WebSocket Gateway | 实时消息推送 |
| Slack | Socket Mode | 企业协作平台 |
| 飞书 | WebSocket 长连接 | 字节跳动企业通讯 |
| 钉钉 | Stream Mode | 阿里企业通讯 |
| QQ | botpy SDK | 腾讯官方 SDK |
| WhatsApp | 第三方 Bridge | 通过 Bridge 接入 |
| Email | IMAP/SMTP | 传统邮件协议 |
| Mochat | Socket.IO | 自定义平台 |

## 六、Cron 定时任务系统

### 6.1 CronService - 定时任务服务

#### 任务定义与存储

定时任务以"渠道:聊天ID:任务名"为唯一标识，存储在 `.cron.json` 文件中。每个任务包含：

- Cron 表达式（定义执行时间）
- 消息内容（要发送给 Agent 的指令）
- 创建时间等元数据

#### 调度执行机制

CronService 使用 croniter 库解析 cron 表达式，计算下次执行时间。后台任务定期检查是否有到期的任务，如果有，构造一个 InboundMessage 并投递到 MessageBus，由 AgentLoop 正常处理。

#### 任务持久化

任务定义被持久化到文件，系统重启后能够恢复所有定时任务。这确保了用户设置的周期性任务（如每天早上发送天气提醒）不会因为系统重启而丢失。

## 七、二次开发 - API Server

### 7.1 FastAPI 服务层

#### 设计目标

API Server 的目标是为前端 Web 界面提供 HTTP 接口，同时复用 Nanobot 核心的所有能力。它作为 Web 前端和 Nanobot 核心之间的桥梁。

#### 启动初始化

服务启动时执行以下初始化步骤：

1. 加载 Nanobot 配置文件
2. 创建 MessageBus 实例
3. 初始化 LLM Provider
4. 创建 AgentLoop 实例
5. 注册所有内置工具

这些组件的生命周期与 API Server 绑定，确保服务就绪时所有依赖都已准备好。

#### 配置热重载

当用户在前端修改配置后，可以调用 `/api/config/reload` 接口触发配置重载。系统会重新读取配置文件、重新创建 Provider 和 AgentLoop，实现配置的即时生效。

### 7.2 API 接口设计

| 端点 | 方法 | 功能 |
|------|------|------|
| /api/chat | POST | 非流式聊天 |
| /api/chat/stream | POST | SSE 流式聊天 |
| /api/sessions | GET | 会话列表 |
| /api/sessions/:id | GET | 会话详情 |
| /api/sessions/:id | DELETE | 删除会话 |
| /api/config | GET | 获取配置 |
| /api/config/reload | POST | 重载配置 |
| /health | GET | 健康检查 |

### 7.3 流式聊天接口

#### SSE 协议应用

流式聊天接口采用 Server-Sent Events（SSE）协议，这是一种服务器向客户端单向推送事件的标准。相比 WebSocket，SSE 更简单、更适合"服务器推送"场景。

#### 事件流设计

| 事件类型 | 数据结构 | 说明 |
|---------|---------|------|
| content | {"type": "content", "content": "..."} | 文本内容块 |
| reasoning | {"type": "reasoning", "content": "..."} | 推理过程 |
| tool_call_start | {"type": "tool_call_start", "name": "..."} | 工具调用开始 |
| tool_call_end | {"type": "tool_call_end", "result": "..."} | 工具调用结束 |
| iteration_start | {"type": "iteration_start", "iteration": 1} | 迭代开始 |
| done | {"type": "done", "content": "..."} | 处理完成 |
| error | {"type": "error", "error": "..."} | 错误 |

#### 会话管理

每次流式请求需要指定 session_key，用于标识会话。同一 session_key 的多次请求共享对话历史，实现连续对话。前端可以为不同的聊天窗口生成不同的 session_key，实现多会话并行。
